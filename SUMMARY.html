The big picture

SAMAGRA is the control room for a one-person physics teaching business.

It does not replace the tools that already work.

It watches them, lines up the work, and makes sure nothing reaches a student without a human "yes".

Local-first Built for one person Read-only by default Quality over speed

01 · The one idea

SAMAGRA owns one thing: the journey of each piece of work.

From a rough idea to something published.

Every tool keeps owning its own content. SAMAGRA just tracks where each item is on its journey, and points at where it lives.

02 · The lifecycle

Every item walks the same four steps.

1
Capture

An idea comes in — often a quick note from the phone.

2
Enrich

An agent drafts it into real teaching material.

3
Review

Another agent checks it. Code and physics are checked separately.

4
Publish

Deepak approves. Always. This step is never automated.

03 · The company

SAMAGRA is run like a small company. The titles are just plain roles.

The "company" is just a way to talk about it. It is not extra software — no special tables, no extra machinery.

04 · Rules we never break

Four lines hold the whole thing together.

Read-only by default

SAMAGRA can write in exactly one place — and only after a human "yes". Everywhere else, it only reads.

One sacred gate

Nothing publishes without Deepak. That approval is never handed to a machine.

Freeze the shape

Big ambitions can be named today. They are not allowed to add anything to build today.

A built-in off-switch

If SAMAGRA isn't clearly saving time, we stop and go back to the old tools. The exit is written down on purpose.

Now (20 Jun): the control room's first floor is built. SAMAGRA OS — an OS-style screen with windows, a dock, and small built-in apps (a clock, notes, a terminal, a calculator-style settings panel, a game) — is up and running on tested foundations. The trick that makes it solid: every piece of real behaviour (how windows move and stack, how the clock and game tick, how the terminal reads commands) lives in small, separately-tested building blocks, so the screen is checked by 166 automated tests, all green, on top of the backend's 102. What's left here is a human look-and-feel pass — a person eyeballing that it looks and feels right, which no test can judge. Next: wire the real tools — pipelines, questions, lectures, the org chart — into the new screen. The single write (a phone note → a draft idea) comes after that.

06 · Decided & parked

What's settled, and what's named-but-not-built.

Decided

  • The headline number is time savedMinutes of Deepak's attention per published item.
  • Reuse the existing topic list~100 tags we already have — never a sprawling map.
  • First self-driving lane is refreshing read-only dataThe safest possible thing to automate — and only later.
  • Published work may go publicThrough a separate front — never SAMAGRA itself.

Parked (named, not built)

  • A coverage scoreboardWhich topics are well-covered, which have holes.
  • Learning from past approvalsSo drafts start closer to Deepak's taste.
  • Self-driving lanesEarned slowly, one safe lane at a time.
  • A "keep going while I'm away" valveFor exam-season crunch, tightly limited.
