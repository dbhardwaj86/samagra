SAMAGRA — Codex Review & Phase-1 Loop Plan

Three independent gpt-5.5 Codex passes (adversarial code review, vision/architecture review, loop planning) reconciled into one execution plan by the CEO.

1 · Overview

Phase 0 (the teachingos → samagra rename) is complete and pushed. Before opening Phase 1, we commissioned three adversarial Codex reviews to establish a baseline. The headline conclusion is a sequencing change: the planned Phase-1 work adds new tests, a new pipeline, and a new state-writing scheduler helper — directly on top of a test suite that overwrites the real database and a state/lock layer that is not atomic. We stabilize that foundation first, then build.

One-line verdict. The spine is clean and the direction is sound if read-only safety becomes enforced, not conventional. Fix the test-harness + atomic state/lock layer first (Track A), ship Phase-1 as read-only clients/adapters (no write methods yet), and harden the HTTP trust boundary in parallel before any non-localhost deploy.

4 · Architecture verdict

The adapter-as-index pattern is the right foundation — SAMAGRA reflects and routes; subsystems keep authority over their own lifecycles. Strengths: simple Artifact contract, FTS5 well within SQLite's comfort zone at 7k rows, conservative bounded scheduler with explicit gates. The weaknesses cluster where the plan is too trusting of convention:

  • Read-only is not enforced — file adapters rely on convention, and the planned McdClient.create_seed punches a write hole before governance exists.
  • One DB, two jobs — samagra.db mixes rebuildable catalog with would-be-durable governance state.
  • Sensitive data duplication — the munshi adapter would copy full payload/person into the catalog.
  • Fail-closed pre-commit with no escape hatch wedges all commits when Codex is offline — and is --no-verify-bypassable anyway, so it isn't real enforcement.
  • No idempotency for the future capture→seed loop → duplicate seeds on retry.

The org metaphor (CEO/COO/Chief Architect, worktrees) should stay operating language and permission labels — not a driver of schema or branch structure. Per-agent worktrees can be deferred until parallel editing is real.

6 · Revised roadmap

Three tracks. Stabilize is fix-first and gates the build loops that depend on it. Build is Phase-1 proper. Harden runs in parallel on a second Khanak instance (disjoint files; one coordination point on scheduler.py).
