Overview

SAMAGRA is an agentic content-pipeline control plane for a JEE/NEET physics teaching operation. It orchestrates several already-working tools — the QX question engine, the physics-textbook lecture engine, booklets, INSP papers and Pratyaksh sims — behind one portal, fills the thin/thick lecture-generation gap, and runs semi-autonomously with hard gates at publish and Telegram + email notifications.

Repo: github.com/dbhardwaj86/samagra · PR #1 squash-merged to main · local-first (Python + FastAPI). Slice 1 is code-complete and verified.

⚠ direction drifting
The 2026-06-19 vision retired the word "OS" ("it silently licenses OS-sized scope") and bound the project to an attention-ROI kill-criterion — yet the project pivoted to a literal 17-app "SAMAGRA OS" and parked the value-producing active loop. The OS experience spec only half-reconciles this — it argues the windowing metaphor is "the honest shape of the work" and firewalls write paths, but never restates the attention-ROI north-star or the kill-criterion, and the value-producing active loop (Phase 3) — the mechanism that actually saves owner attention — is parked while OS chrome (3 themes, a Snake game, mobile mode) grows. The "frugal" first-class value sits unaddressed against a 17-app GUI. Full reviews: docs/superpowers/_research/samagra-os/_vision-review-output.md (+ _vision-review-prompt.md, _vision-review.log).

Decision (ratified 2026-06-21 by Deepak — now binding):

  1. DEC-1 · Scope. SAMAGRA OS is a bounded operator console — a UI metaphor only; SAMAGRA remains a control plane and does not acquire app-platform scope.
  2. DEC-2 · North-star binding. The attention-ROI north-star + the kill-criterion remain binding and are not voided by the OS track (data source = the governance events/review_overlay ledger; the ~3 hrs/wk figure stays the seed proposal pending the DEC-4 gauge).
  3. DEC-3 · Scope firewall (hard non-goal, mirrored into OS spec §3): no entertainment apps beyond E1's Snake, no third-party apps / marketplace, no process-scheduler model, no user-facing product identity.
  4. DEC-4 · Attention-ROI acceptance gate before E3: 2–3 representative operator tasks (triage munshi captures · read pipeline/gate status · find + open a catalog artifact) measured for owner time-saved vs the old portal must pass before any E3 work; fail → freeze GUI expansion + reprioritize Phase 3.
  5. DEC-5 · Phase 3 is the primary value engine — the active loop restarts after the E2 visual-QA pass and the DEC-4 gate, ahead of further theme/mobile polish.
